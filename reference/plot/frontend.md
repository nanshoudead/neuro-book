# 剧情模块前端 / UIUX 规范 v0.1

## 一、目标

本稿用于约束剧情模块在前端中的产品形态与交互方式。

它要解决的问题不是“数据库如何设计”，而是：

- 作者如何快速定位一条 `Thread`
- 作者如何规划一条 `Thread` 下的 `Scene`
- 作者如何规划某一章的内容
- 作者如何直观看到多线剧情的层次结构

当前版本明确以以下对象为核心：

- `Thread`
- `Scene`
- `Chapter`
- `Plot`

其中：

- `Story`、`StoryPhase` 主要承担定位、分组、检索作用
- `Thread`、`Scene`、`Chapter`、`Plot` 才是前端高频操作中心

---

## 二、总设计原则

### 1. 剧情模块必须成为主工作区，不是附属弹窗

剧情系统是产品核心功能，因此不应只塞进一个窄侧栏。

推荐形态：

- 左侧仍保留 IDE 一级导航
- 现有 `outline` 入口应升级为正式的 `plot` / `剧情`
- 进入剧情模块后，中央主工作区切换为剧情工作区
- 右侧 Agent 抽屉仍可保留，用于围绕当前 `Thread / Scene / Chapter` 协作

也就是说：

- `章节编辑器` 是一个主工作区
- `剧情工作区` 也是一个主工作区

两者地位应当平级。

---

### 2. 四个视图不是四个独立系统，而是同一工作区的四种观察方式

这四个视图必须共享同一套选择状态，而不是彼此割裂：

- 当前 `Story`
- 当前 `Phase`
- 当前 `Thread`
- 当前 `Scene`
- 当前 `Chapter`

推荐原则：

- 顶部使用视图切换器
- 切换视图时，尽量保留当前选择上下文
- 让用户感觉是在“切换看法”，而不是“跳到另一个产品”

---

### 3. 视图职责必须单一

每个视图都只服务一个主要目标：

- 视图一：定位 `Thread`
- 视图二：规划 `Thread`
- 视图三：规划 `Chapter`
- 视图四：理解整体层次与多线结构

不允许一个视图同时塞进太多目标，否则会退化成杂乱的大面板。

---

### 4. 主编辑单位必须稳定

虽然有四个视图，但默认编辑重心应稳定在：

- `Thread`
- `Scene`
- `Plot`

其中：

- `Story`、`StoryPhase` 主要用于看和筛
- `Chapter` 主要用于编排正文承载

---

## 三、信息架构

### 1. 一级入口

建议把当前 IDE 左侧的 `outline` 入口改为：

- `plot`
- 中文显示为“剧情”

原因：

- `outline` 过于泛，容易和传统大纲混淆
- 当前系统实际上已经不是简单大纲，而是完整剧情模块

---

### 2. 剧情工作区总布局

推荐采用三段式布局：

```text
左：导航 / 筛选 / 树
中：主视图画布
右：详情检查器 / 引用 / 写作提示
```

其中：

- 左侧负责“找对象”
- 中间负责“看结构、做拖拽、做规划”
- 右侧负责“编辑当前选中对象”

推荐宽度倾向：

- 左栏：280 - 320
- 中栏：自适应主空间
- 右栏：320 - 380

这比把所有信息堆在一个列表里更适合剧情系统。

---

### 3. 顶部视图切换器

剧情工作区顶部需要固定视图切换器。

推荐四个入口：

1. `线索`
2. `线程`
3. `章节`
4. `树图`

推荐对应关系：

- `线索` = 视图一：`Story / Phase / Thread`
- `线程` = 视图二：`Thread / Scene`
- `章节` = 视图三：`Chapter / Plot`
- `树图` = 视图四：类 git tree 图

不建议直接把按钮名写成“视图一 / 视图二”。

---

## 四、四个核心视图

### 1. 视图一：`Story / Phase / Thread` 视角

### 1.1 目标

此视图的唯一主要目标是：

- 便于快速定位到某条 `Thread`

它不是主要编辑视图。

---

### 1.2 推荐布局

推荐三列：

```text
左列：Story 与 Phase 列表
中列：Thread 列表
右列：Thread 摘要卡 / 快速统计
```

左列内容：

- 当前 `Story` 摘要
- `StoryPhase` 列表
- `未分组线程` 入口

中列内容：

- 当前 `Phase` 下所有 `Thread`
- 支持搜索、标签过滤、主线过滤、状态过滤

右列内容：

- 当前选中 `Thread` 的摘要
- `Scene` 数量
- 已挂入 `Chapter` 的 `Scene` 数量
- 最近更新时间
- `refs` 数量
- `writingTip`

---

### 1.3 关键交互

- 点击 `Phase`：更新中列 `Thread` 列表
- 点击 `Thread`：右侧显示摘要，并允许进入视图二
- 双击 `Thread` 或点击“规划线程”：跳到视图二
- 搜索范围默认仅当前 `Story`

---

### 1.4 为什么需要这个视图

因为当项目规模变大后，作者最先遇到的问题通常不是“怎么编辑 Scene”，而是：

- 我现在要找哪条线
- 这条线在哪个阶段
- 哪条线是主线
- 哪条线已经暂停太久

这个视图是“定位器”，不是“工地”。

---

### 2. 视图二：`Thread / Scene` 视角

### 2.1 目标

此视图用于：

- 规划一条 `Thread`
- 编排该 `Thread` 下的 `Scene`

它应当成为剧情模块的主编辑视图。

---

### 2.2 推荐布局

推荐两段主区 + 右侧检查器：

```text
顶部：Thread 头部信息
中部：Scene 序列画布
右侧：当前 Scene 详情编辑
```

顶部显示：

- `Thread.title`
- `isMainThread`
- `status`
- `tags`
- `summary`

中部显示：

- 当前 `Thread` 下的 `Scene` 卡片序列
- 默认按 `threadSortOrder` 排列
- 适合横向时间线或纵向卡片流

每个 `Scene` 卡片建议包含：

- 标题
- 简短 `summary`
- `purpose`
- 关联 `Chapter`
- `Plot` 数量
- `refs` 数量
- 当前状态

右侧检查器建议编辑：

- `Scene.title`
- `summary`
- `purpose`
- `status`
- `chapterId`
- `writingTip`
- `refs`
- `plots`

---

### 2.3 关键交互

- 拖拽 `Scene` 卡片：修改 `threadSortOrder`
- 在 `Scene` 卡片上快速新建前后场景
- 点击 `Scene`：右侧加载详情
- 点击 `Scene` 的 `Chapter` 标签：跳到视图三对应章节
- 卡片上直接展示 `effectiveRefs` 数量，但不要把完整 ref 文本挤在卡片面上

---

### 2.4 视觉建议

此视图应更偏“卡片墙 / 时间线”，而不是纯表格。

原因：

- `Thread` 本质是线性的因果组织
- `Scene` 更适合通过卡片理解顺序、缺口、密度

如果直接做成表格，会失去“规划感”。

---

### 3. 视图三：`Chapter / Plot` 视角

### 3.1 目标

此视图用于：

- 规划一章的内容
- 理解一章内部承载了哪些 `Scene` 与 `Plot`

这里的主问题不是“这条线怎么走”，而是：

- 这一章到底写什么
- 节奏如何安排
- 爽点、冲突、反转、悬念落在哪里

---

### 3.2 推荐布局

推荐三栏：

```text
左列：Chapter 列表
中列：当前 Chapter 内容结构
右列：当前 Plot / Scene 详情
```

左列：

- 仍然沿用现有 `Volume / Chapter` 浏览习惯
- 但聚焦 `Chapter`

中列：

- 显示当前 `Chapter` 下的 `Scene` 列表
- 每个 `Scene` 下展开其 `Plot`
- 即：

```text
Chapter
  ├── Scene A
  │     ├── Plot 1
  │     ├── Plot 2
  ├── Scene B
        ├── Plot 3
```

右列：

- 编辑选中 `Scene` 或 `Plot`
- 如果选中 `Plot`，重点编辑：
  - `kind`
  - `summary`
  - `effect`
  - `writingTip`

---

### 3.3 为什么这里是 `Chapter / Plot`，而不是 `Chapter / Scene`

因为这个视图的目标是规划一章内部的节奏承载。

而节奏通常更接近：

- 铺垫
- 冲突
- 反转
- 揭示
- 爽点
- 结果

这些更直接对应 `Plot`。

但界面中仍然要保留 `Scene` 作为分组层，否则 `Plot` 会失去上下文。

所以它的真实结构应该是：

```text
Chapter -> Scene -> Plot
```

只是用户心智目标是“我在规划这一章的 Plot 节奏”。

---

### 3.4 关键交互

- 左侧切换 `Chapter`
- 中部可拖拽 `Scene` 调整 `chapterSortOrder`
- `Scene` 内部可拖拽 `Plot`
- 点击某个 `Plot.kind` 标签，支持按 `despair / relief / reward / conflict / twist / mystery` 快速筛看

---

### 4. 视图四：类 git tree 图视角

### 4.1 目标

此视图用于：

- 直观理解 `Story / Phase / Thread / Scene` 的层级
- 理解多线剧情的并行展开
- 从整体上看哪里线太密、哪里线中断

这是认知视图，不是高频编辑视图。

---

### 4.2 推荐形态

推荐采用“类 git graph + tree”混合视图，而不是传统文件树。

也就是：

- 左侧仍然有层级缩进
- 但在线程与场景节点之间，用线段表示分支关系

建议节点层级：

```text
Story
  ├── Phase A
  │     ├── Thread A1
  │     │     ├── Scene A1-1
  │     │     ├── Scene A1-2
  │     ├── Thread A2
  │           ├── Scene A2-1
  ├── Phase B
```

但视觉上应更强调：

- 分支
- 并行
- 深度

---

### 4.3 节点视觉建议

- `Story`：最大节点，单一头节点
- `Phase`：二级分组条
- `Thread`：分支主节点，颜色最明显
- `Scene`：小节点，密度高但信息简洁

推荐：

- 同一 `Thread` 使用统一色带
- `Scene` 节点继承所属 `Thread` 的颜色
- 主线 `Thread` 用更高对比度，不靠额外图标硬堆

---

## 五、时间轴 Preview 原型

### 1. 目标

时间轴 Preview 是独立于主剧情工作区的原型视角。

它的唯一目标是：

- 把单个 `StoryPhase` 下的 `Thread / Scene` 编排成正文展示顺序
- 直观看出多条 `Thread` 在正文章节中的并行关系

它不是树图，也不是章节树。

---

### 2. 当前原型规则

首版固定采用以下约束：

- 独立 preview 页面，不并回主工作区
- 一次只看一个 `StoryPhase`
- 横轴时间定义为正文顺序
- 每条横向泳道代表一条 `Thread`
- 横向按 `Scene` 槽位排布，不按真实时长
- 同一个横向槽位允许不同 `Thread` 并行出现 `Scene`
- `Chapter` 通过背景分段表达，不额外占用泳道
- 未挂章 `Scene` 统一进入末尾的“草稿尾区”
- 首版不做拖拽，只做浏览、定位与检查器联动

---

### 3. 推荐布局

推荐沿用独立 preview 的三段式布局：

```text
上：StoryPhase 切换 / 控制条
左中：横向时间轴主画布
右：检查器
```

主画布本身采用：

```text
顶部：Chapter 背景分段
左侧：Thread 泳道标题
中间：Scene 固定宽度卡片
末尾：草稿尾区
```

---

### 4. 关键视觉与交互

- 主线泳道优先显示，并保留更强强调色
- `Scene` 卡片使用固定宽度，不做时长缩放
- 点击 `Thread`：右侧检查器切到该线
- 点击 `Scene`：右侧检查器切到该场景
- 点击 `Chapter` 分段：检查器切到该章，但不强制清空当前 `Thread / Scene`
- 横向滚动必须稳定，顶部章节段与内容区需要保持同一滚动上下文

---

### 5. 与其他视图的区别

- `Thread / Scene` 视图更像因果规划器
- `Chapter / Plot` 视图更像章节节奏编辑器
- 时间轴 Preview 更像正文编排器

也就是说：

- 树图回答“线怎么分叉”
- 线程视图回答“线内部怎么推进”
- 时间轴回答“这些线如何进入正文顺序”

---

## 六、单 Thread 侧边栏 Preview

### 1. 目标

单 Thread 侧边栏 Preview 用于验证一种更窄、更聚焦的局部规划形态：

- 一次只看一条 `Thread`
- `Scene` 从上到下排列
- 用底部 detail 面板补足当前 `Scene` 的结构化信息

它不是完整剧情工作区，而是“侧边栏局部工作视角”。

---

### 2. 容器约束

当前原型明确采用和 `NovelIdeToolPanel` 相同的容器语义：

- 固定侧栏宽度
- 上方工具头
- 中间主列表
- 底部详情面板

但它只是独立 preview，不直接接入真实 `NovelIdeToolPanel` 的 tab 分发逻辑。

---

### 3. 当前原型规则

- 只展示单条 `Thread`
- 允许在工具区切换不同 `Thread`
- 当前 `Thread` 下的 `Scene` 按 `threadSortOrder` 自上而下排列
- 主列表只负责定位和浏览
- 底部 detail 面板负责展示 `summary / purpose / chapter / plots / refs / writingTip`
- detail 面板可以展开、收起、最小化、拖拽高度
- 首版不做拖拽、不做保存、不做跨 Thread 操作

---

### 4.4 关键交互

- 点击 `Thread`：跳到视图二
- 点击 `Scene`：右侧查看详情或跳到视图二对应位置
- 缩放层级：可切换“只看 Thread” / “展开到 Scene”
- 支持按 `Phase` 折叠

---

## 五、共享交互规则

### 1. 统一对象高亮

无论在哪个视图，只要选中了某个对象：

- 左侧导航
- 中部主视图
- 右侧检查器

都应同步高亮同一个对象。

---

### 2. 统一跳转规则

推荐跳转逻辑：

- 从视图一点击 `Thread` -> 视图二
- 从视图二点击 `Chapter` 标签 -> 视图三
- 从视图四点击 `Thread / Scene` -> 对应视图二

不要让用户自己重新找一遍。

---

### 3. 右侧检查器保持一致

四个视图的右侧检查器应尽量复用一套结构：

- 基础信息
- `summary`
- `note`
- `writingTip`
- `refs`

只是根据对象类型调整字段。

这样用户不会在四个视图里学四套编辑心智。

---

### 4. 空状态必须指导下一步动作

剧情系统会大量出现空状态，例如：

- 还没有 `Thread`
- 某个 `Thread` 还没有 `Scene`
- 某个 `Chapter` 还没有挂接 `Scene`

空状态不应只写“暂无数据”，而应明确下一步：

- 新建线程
- 新建场景
- 把当前 Scene 挂到某章
- 为当前 Scene 添加第一个 Plot

---

## 六、与现有 IDE 的整合建议

### 1. 导航层

建议修改当前：

- `outline` -> `plot`

中文标题从“剧情大纲”升级为“剧情系统”或“剧情”。

---

### 2. 主工作区层

当前中央区域默认是章节编辑器。

建议升级为两种主模式：

- `chapter-editor`
- `plot-workspace`

不要把剧情系统塞进现有 `340px` 工具侧栏里，否则会直接变成次要功能。

---

### 3. Agent 协作层

右侧 Agent 抽屉应能够感知当前剧情上下文。

建议后续支持：

- 当前选中 `Thread`
- 当前选中 `Scene`
- 当前选中 `Chapter`

这样 Agent 才能真正围绕剧情工作区协作。

---

## 七、MVP 落地顺序

为了降低实现风险，建议按以下顺序落地：

### 第一步

先完成基础工作区骨架：

- `outline` 改为 `plot`
- 中央新增剧情工作区
- 顶部四视图切换器
- 左中右三栏布局

---

### 第二步

优先做视图一 + 视图二：

- 视图一解决“找 Thread”
- 视图二解决“规划 Thread”

这是剧情模块最核心的价值闭环。

---

### 第三步

再做视图三：

- 把 `Chapter -> Scene -> Plot` 的章级规划视图补上

这一步才真正把剧情和正文编排联起来。

---

### 第四步

最后做视图四：

- 类 git tree 图

原因：

- 它最有展示价值
- 但不是最先解决生产效率的问题

---

## 八、一句话总结

> 剧情模块前端不应只是“剧情大纲侧栏”，而应成为与章节编辑器平级的主工作区。  
> 四个视图共享同一套上下文：视图一负责找 `Thread`，视图二负责规划 `Thread`，视图三负责规划 `Chapter`，视图四负责理解多线结构。  
> 真正的高频编辑中心应始终落在 `Thread / Scene / Chapter / Plot` 上。
