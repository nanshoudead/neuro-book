# 世界模拟

> 兼容入口：新的标准 workflow skill 是 `novel-workflow-06-emulation-tick`。后续写作模式中推进一个世界运行 tick、产出 `simulation/runs/ticks/{id}-{slug}/report.md` 和 `prose.md` 时，优先读取并遵守该 skill。本文件保留为旧中文入口和方法补充。

用于把静态世界书转换成“当前这一刻发生了什么”。它服务开局剧情设计和后续写作，不替代剧情规划。当前落地目录仍是 `simulation/`，写作流程中可称为 emulation。

推演不是 Agent 一个人的事。用户是主导者，用户可以提供“因”、初始变量、想观察的区域或想施加的压力；Agent 负责根据已确认设定推导过程和结果。缺少关键起因时，先给用户几个候选起因或请用户指定，不要独自定死会影响主线的重大因果。

## 前置要求

开始前应尽量具备：

- 小说基础定位和简介。
- 粗略剧情种子或当前阶段目标。
- 世界书初始化产出的关键角色、势力、地点、规则。

如果关键世界规则仍然缺失，先回到“世界书初始化流程”。

## 执行顺序

### 1. 确定模拟切片

默认只模拟当前开局或当前阶段需要的状态，不推演整本书。但可以同时选择多个切片，从不同尺度观察同一时间段。

先明确：

- 当前时间点。
- 主角所在位置。
- 主角短期目标。
- 正在逼近的威胁或机会。
- 用户提供的起因或初始变量。
- 本轮要模拟哪些切片。

常用切片：

- 主角视角：主角能看到、感受到、误判和选择的东西。
- 关键人物视角：重要人物正在做什么，他们掌握哪些信息。
- 势力视角：组织、国家、家族、宗门、公司或阵营如何行动。
- 地点/生态视角：地点秩序、资源、环境、危险和普通生活如何变化。
- 世界范围自然运转：不围着主角转的自然变化、资源流动、灾害、贸易、流言和制度惯性。

### 2. 推演多条因果链

每次世界模拟至少选择 2 到 3 条与当前剧情有关的因果链。因果链可以来自资源、环境、势力、人物、制度、灾害或异常规则。

推演格式：

1. 起因：用户给出的因，或从已确认设定中选出的初始变量。
2. 过程：按规则、资源、人物意图、势力反应、环境变化逐步推出后果。
3. 结果：当前时间点已经发生什么，谁知道，谁还不知道。
4. 影响：它如何改变角色状态、势力态势、地点压力或短期任务。
5. 观察点：尚未爆发但值得继续追踪的变量。

示例链条：大量开采魔灵矿 -> 当地自然条件恶化 -> 灾难或污染迹象出现 -> 矿场、领主、商会、普通居民和主角相关势力分别反应 -> 当前地点状态、势力行动和主角可见线索发生变化。

不要把因果链直接写成剧情大纲。它只解释世界怎样自然运转，以及这些变化给当前剧情施加什么压力。

### 3. 加入随机性扰动

模拟过程中可以加入少量随机性变化，让世界不完全按主角或势力计划运行。

随机性可以来自：

- 天气、地质、疾病、灾害、野兽、生态异常。
- 误判、流言、意外目击、信件延误、交易失败。
- 资源价格波动、矿脉异动、仪式副作用、规则失控。
- 某个小人物的选择、恐惧、贪婪、善意或失误。

随机扰动可能没什么作用，也可能成为后续重大事件的种子。写入时标成“当前迹象”“潜在变量”或“后续观察点”，不要直接宣布它一定会改变主线。重大随机结果在影响主线前，应继续让用户主导确认。

### 4. 设置角色状态

为主角和开局关键人物更新 subject 文件：

- `events.md`：他们经历、观察、听说或被告知了什么。
- `knowledge.md`：他们已经知道、相信或误解了什么。
- `mind.md`：他们当前想法、判断、疑虑和动机。
- `state.md`：GM 裁决后的当前处境、已知目标、可用资源、关系压力和状态快照。

### 5. 设置势力与环境状态

按需更新势力、地点、规则或实体状态：

- 势力当前行动、误判、追捕、交易或冲突。
- 地点当前危险、秩序、资源、限制。
- 环境压力，例如灾害、封锁、竞赛、倒计时。
- 随机扰动留下的迹象、潜在变量或后续观察点。
- 需要状态追踪的特殊实例写入 `simulation/entities/{entity-id}/`；普通可堆叠物品只写 subject `state.md` 摘要。

### 6. 形成开局任务

把粗略剧情种子压成 1 到 3 个当前任务：

- 主角眼前必须解决的问题。
- 读者能理解的短期目标。
- 会把主角推向下一场景的外部压力。

不要在这里拆 Thread / Scene / Plot；那属于“开局剧情设计”或“剧情规划流程”。

## 写入原则

- 稳定设定写 `lorebook/*/index.md`，subject 视角事件写 `simulation/subjects/*/events.md`，真实实例状态写 `simulation/entities/*/state.md`。
- Tick 过程优先写入 `simulation/runs/ticks/{id}-{slug}/report.md`；如有用户可见正文、RP 正文或试写片段，单独写入同目录 `prose.md`。
- `prose.md` 只保存用户最终看到的正文，不保存后台裁决、隐藏真相或工具流水账。
- 状态要短、具体、能被下一步剧情直接使用。
- 不要把未来剧情结果写成当前状态。
- 不要推演用户尚未确认的重大设定。
- 用户提供的起因和本轮推演出的结果要区分清楚。
- 随机扰动只记录为当前迹象、潜在变量或观察点，除非用户确认它已经成为事实。
- 不要把上帝视角 lorebook 直接写入 subject knowledge；subject 只能知道自己经历、被告知、观察或自然推断到的信息。

## 后续衔接

完成后默认进入“开局剧情设计”，用当前状态设计前三章附近的开局推进。若用户要规划更长阶段，再进入“剧情规划流程”。

## 完成标准

- 主角当前处境、短期任务和压力已经明确。
- 已按多个模拟切片观察当前状态。
- 已推演多条因果链，并区分用户提供的因、推演过程和当前结果。
- 随机扰动已被记录为可追踪变量，没有直接伪装成确定主线。
- 关键人物、势力、地点或环境的 `state.md` 能支撑开局剧情。
- 状态没有扩写成完整大纲。
- 下一步可以直接设计开局剧情。
