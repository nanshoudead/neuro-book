# 文本润色

## 工作流程

### 步骤 1：获取输入文本

用户可以通过两种方式提供文本：

**方式 A：提供文件路径**
- 用户："帮我润色 manuscript/chapter-01.md"
- 直接使用该文件路径

**方式 B：直接粘贴文本**
- 用户粘贴一段文本
- 写入临时文件：`write(".agent/polish-input.md", 文本内容)`
- 使用 `.agent/polish-input.md`

### 步骤 2：CLI 静态检查

执行 CLI 检查工具：

```bash
bun .nbook/agent/skills/anti-ai-slop/cli/checker.ts check <文件路径>
```

CLI 会输出类似 eslint 的格式化报告，按规则分组展示候选问题：
- 每个问题包含：行号、列号、命中上下文、命中位置指示
- 每条规则附带通用修复建议
- Static rule 只代表“可稳定定位的候选”，不代表必须修改

### 步骤 3：LLM 深度审查 + 快速审查评分

步骤 3 包含三个连续动作：获取 LLM 规则、逐条全文审查、快速审查并评分。不要拆成独立步骤。

先执行：

```bash
bun .nbook/agent/skills/anti-ai-slop/cli/checker.ts show-llm-rules
```

将输出作为本轮全文语义审查的规则来源。如果输出显示当前没有启用 LLM 规则，则跳过 LLM rule 全文审查，但仍要复核 CLI 静态命中项并完成快速审查评分。

**判断原则**：
- **检测结果不一定要修复**：根据实际语境判断。
- **考虑文本类型**：小说、对白、技术文档、报告、营销文案的自然表达不同。
- **尊重作者意图**：如果是刻意的角色声音、讽刺、引用或体裁要求，应保留。
- **偏向具体和直接**：能用具体动作、事实、主体替代抽象评价时，优先建议替代。

**Static Rules 处理**：

1. **High 级别**：强烈建议修复，但仍需读取上下文确认；如果是技术文档的步骤说明，可能应该保留。
2. **Medium 级别**：读取原文上下文（前后 2-3 行），判断是否真的需要修复；如"其实"在对话中可能是自然口语。
3. **Low 级别**：默认保留，除非明显影响阅读密度、自然度或可信度。

**LLM Rules 全文审查**：

1. 对 `show-llm-rules` 输出的每条规则逐条审查原文。
2. 不要等待 CLI 提供 LLM rule 命中项；LLM rule 需要 Agent 主动阅读全文寻找候选。
3. 每条规则都要给出结果：未发现候选、建议修复、建议保留、需要用户确认。
4. 对候选文本读取上下文（前后各 5 行，或整个段落），根据规则的判断标准和示例判断。
5. 如遇到专有名词/科幻设定/同人梗/科学、领域知识/历史事实等不熟悉的领域：
     ```
     invoke_agent("researcher", query: "...")
     ```
     根据调研结果辅助判断。

**快速审查清单**：

- 是否有不增加信息的开场、强调、铺垫或自我说明？
- 是否有“不是 X，而是 Y”“问题不是 X，是 Y”等制造反转的结构？
- 是否用反问、设问、三段式排比或金句感结尾代替了具体内容？
- 是否有抽象评价，例如“意义重大”“影响深远”，但没有说清发生了什么？
- 是否隐藏了行动者，例如“决定产生了”“局面发生变化”，却没有说明谁做了什么？
- 是否过度解释读者已经能理解的内容，削弱信任感？
- 是否连续几句长度、结构、段尾节奏过于接近？
- 是否使用懒惰绝对词，例如“所有人”“永远”“一定”，但没有具体限定？

**50 分评分**：

| 维度 | 问题 | 分值 |
|------|------|------|
| Directness | 是直接陈述，还是先公告、铺垫、绕弯？ | 1-10 |
| Rhythm | 句长、段落、收束方式是否有变化？ | 1-10 |
| Trust | 是否信任读者，避免说教、反复解释和情绪操控？ | 1-10 |
| Authenticity | 是否像具体作者/角色/场景发出的声音？ | 1-10 |
| Density | 是否存在可删掉而信息不变的词句？ | 1-10 |

评分处理：
- **低于 35/50**：必须进入重点重写建议，优先处理结构、节奏和具体性。
- **35-42/50**：做局部修订，优先删除填充、重写空泛句和调整段落节奏。
- **43/50 以上**：只做轻微润色建议，避免把作者风格磨平。

### 步骤 4：生成修复计划（用户审批）

生成修复计划文档：`.agent/polish-plan.md`

内容包括：
- **统计**：强烈建议修复多少项、建议修复多少项、保留多少项、不确定多少项
- **评分**：Directness / Rhythm / Trust / Authenticity / Density 五项分数和总分
- **修复详情**：每条修改的原文、修改后、理由
- **不确定项**：需要用户确认的问题，附带建议和疑问

### 步骤 5：执行修复

按用户审批后的计划执行修复。修改文本时优先生成 `.agent/polish-output.md`，避免直接覆盖原文件；如用户明确要求改原文件，再按从后往前的顺序应用编辑，减少行号偏移。

### 步骤 6：生成修复报告

生成最终报告。

内容包括：
- **统计**：总候选数、已修复、保留不改、修复类别分布
- **评分变化**：如果能评估，说明润色前后的 50 分评分变化
- **修复详情**：每条修改的 diff（- 原文 / + 修改）
- **未修复项**：保留的问题及原因
- **输出文件位置**

展示给用户：
- 简要统计数字
- 输出文件位置
- 可选：展示几个关键修复的 diff 示例

## CLI 工具使用说明

**检查文件**：
```bash
bun .nbook/agent/skills/anti-ai-slop/cli/checker.ts check <file>
```

**显示 LLM 规则**：
```bash
bun .nbook/agent/skills/anti-ai-slop/cli/checker.ts show-llm-rules
```

**兼容旧用法**（不推荐）：
```bash
bun .nbook/agent/skills/anti-ai-slop/cli/checker.ts <file>
```

退出码：
- `0`：未发现问题或只有 low/medium 级别
- `1`：发现 high 级别的问题

## 注意事项

1. **临时文件管理**：所有临时文件统一放在 `.agent/` 目录
2. **错误处理**：CLI 工具失败时，检查文件路径和 bun 环境
3. **编码问题**：确保文件是 UTF-8 编码
4. **行号偏移**：修改时从后往前进行，避免前面的修改影响后面的行号
5. **不确定项处理**：遇到拿不准的问题，优先标记为需要用户确认，不要强行修复
6. **风格保护**：不要为了消除套路化表达，把有效的人物声音、文体特征或作者节奏改平
