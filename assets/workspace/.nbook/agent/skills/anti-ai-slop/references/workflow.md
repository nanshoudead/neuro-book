# 完整流程详解

本文档深入解释"去 AI 味道"skill 的 6 步润色流程，包括每步的决策逻辑、边界情况处理和错误恢复策略。

## 流程概览

```
用户输入 → CLI 静态检查 → LLM 深度审查 → 生成修复计划（用户审批）→ 执行修复 → 生成报告
```

## 步骤 1：获取输入文本

### 目标
将用户提供的文本转换为可检查的文件路径。

### 实现细节

**识别输入类型**：
```typescript
if (用户消息包含文件路径) {
  // 识别模式：
  // - "检查 manuscript/chapter-01.md"
  // - "帮我看看 .agent/test.md 的 AI 味道"
  文件路径 = 提取的路径
} else {
  // 用户直接粘贴文本
  文件路径 = ".agent/polish-input.md"
  write(文件路径, 用户粘贴的文本)
}
```

### 边界情况

**文件不存在**：
- 检查文件路径是否有效
- 如果无效，询问用户确认路径

**路径歧义**：
- `manuscript/chapter-01.md` 可能指多个 workspace
- 确认当前 workspace 上下文

**空文本**：
- 用户粘贴空文本
- 提示用户提供有效内容

### 错误恢复

- 文件读取失败 → 提示用户检查路径和权限
- 编码问题 → 提示确保 UTF-8 编码

---

## 步骤 2：CLI 静态检查

### 目标
快速识别可以稳定定位的候选问题，生成可供 Agent 复核的报告。

### 实现细节

**执行命令**：
```bash
bun assets/workspace/.nbook/agent/skills/anti-ai-slop/cli/checker.ts check <文件路径>
```

**解析输出**：
CLI 输出按 static rule 分组，包含规则 ID、命中上下文、行号、列号和修复建议：
```text
filler-word-actually (其实、实际上、事实上)
  1:9  这个问题很复杂。其实我们可以从另一个角度来看。
                 ^^

  1 occurrence. 修复建议：这类填充词通常不增加实质内容，建议直接删除。
```

注意：CLI 只负责定位候选文本，不负责决定是否修复。

### 边界情况

**CLI 执行失败**：
- bun 未安装 → 提示用户安装 bun
- 文件路径错误 → 检查路径是否正确
- 规则文件损坏 → 检查 JSON 文件是否有效

**无问题**：
- CLI 输出"✓ 未发现问题"
- 告知用户文本没有明显的 AI 味道

**问题过多**：
- 超过 50 个问题
- 建议分段处理或降低检查严格度

### 错误恢复

- CLI 退出码非 0 → 不代表失败，可能只是发现了 error 级别问题
- 输出解析失败 → 直接展示原始 CLI 输出给用户
- 超时 → 文件可能太大，建议分段检查

---

## 步骤 3：LLM 深度审查

### 目标
读取 LLM 规则，并对 static 命中项与 LLM 语义规则进行上下文审查，决定是否真的需要修复。

### 获取 LLM 规则

必须先执行：

```bash
bun assets/workspace/.nbook/agent/skills/anti-ai-slop/cli/checker.ts show-llm-rules
```

这一步仍属于 LLM 深度审查的一部分，不单独拆成新步骤。`show-llm-rules` 的输出是本轮全文语义审查的规则来源。

### 审查流程

1. 复核 CLI static 命中项：
   - 读取命中位置上下文
   - 判断候选是否真的影响文本自然度
   - 给出建议修复、建议保留或需要用户确认

2. 逐条审查 LLM rules：
   - 对 `show-llm-rules` 输出的每条规则主动阅读全文
   - 不等待 CLI 给出 LLM 命中项
   - 找到候选后读取上下文，应用该规则的判断标准和示例
   - 如果没有候选，也在计划中说明未发现明显问题

### 判断标准

**Static Rules**：
- High：强烈建议修复，但仍需确认上下文是否合理
- Medium：读取前后 2-3 行判断；对话和口语场景可能保留
- Low：默认保留，除非明显影响阅读

**LLM Rules**：
- 严格遵循 `show-llm-rules` 输出中的判断标准
- 参考示例，但不要机械套用示例
- 不确定时归入“需要用户确认”

### 读取上下文策略

**简单问题（filling words）**：
- 前后各 1-2 行足够

**二元对比和问题定义对比**：
- 这些属于 static rule，因为候选文本可稳定识别
- 修复决策仍需读取前后各 3-5 行，或整个段落

**段落级 LLM rule**：
- 优先读取完整段落
- 如果段落承接前后文，再读取相邻段落

### Web 调研触发

**触发条件**：
1. 文本包含专有名词（人名、地名、作品名）
2. 涉及科幻设定（需要确认是否是原著梗）
3. 同人作品引用
4. 特定领域术语

**调研内容**：
```typescript
await invoke_agent("researcher", {
  query: "《三体》中「不是XX，而是YY」是否是原著名句"
});
```

**使用调研结果**：
- 如果是原著引用 → 保留
- 如果是常见用法 → 根据上下文判断
- 如果查无此内容 → 按 AI 味道处理

### 决策记录格式

审查结论是给用户看的，使用 Markdown 小节、短段落和列表，不使用 JSON、YAML、TypeScript interface 等格式化数据结构。

```markdown
### 行 34:5 - 二元对比结构

原文：不是因为天气不好，而是因为路况复杂。

建议：改为“路况复杂导致延误。”

理由：前半句没有纠正常见误解，只是在制造反转；直接陈述原因更自然。
```

### 边界情况

**问题文本难以定位**：
- 正则匹配到的文本可能不唯一
- 使用行号 + 列号精确定位

**上下文不足**：
- 问题在文件开头/结尾
- 尽可能读取可用的上下文

**循环依赖**：
- 修复 A 可能引入新问题 B
- 第一版不处理，留给后续迭代

### 错误恢复

- LLM 判断超时 → 标记为 `needsUserConfirm`
- Web 调研失败 → 按无调研结果处理
- 上下文读取失败 → 使用问题本身的文本

## 步骤 4：生成修复计划（用户审批）

### 目标
生成清晰的修复计划，让用户理解和审批。

### 计划内容

**统计部分**：
```markdown
## 统计
- 强烈建议修复：X 项
- 建议修复：Y 项（经 LLM 判断，建议修复）
- 保留不改：Z 项（经判断，不需要修复）
- 不确定项：N 项（需要你确认）
```

**修复详情**：
```markdown
### [1] 行 5:12 - 填充词：其实 ✓ 强烈建议修复
**原文**：这个问题很复杂。其实我们可以从另一个角度来看。
**修改后**：这个问题很复杂。我们可以从另一个角度来看。
**理由**：填充词不增加实质内容

### [2] 行 34:5 - 二元对比结构 ❓ 需要确认
**原文**：不是因为天气不好，而是因为路况复杂。
**建议修改**：路况复杂导致我们没有按时到达。
**理由**：可以直接陈述后半部分，但可能损失对比强调的效果
**需要确认**：你认为这里需要保留对比吗？

### [3] 行 67:10 - 二元对比结构 ✗ 保留
**原文**：问题不是技术不够先进，而是沟通不够充分。
**理由**：这个对比纠正了常见误解（以为是技术问题），保留合理
```

### 展示方式

使用面向用户的 Markdown 直接展示修复计划。问题较少时展示完整详情；问题较多时先展示统计、关键样例和不确定项，并把完整计划写入 `.agent/polish-plan.md`。

### 解析用户反馈

**识别模式**：
```
"确认" / "全部接受" / "OK"
  → 接受所有修复

"跳过第 3、5 项"
  → 拒绝第 3 和第 5 项修复

"第 3 项改成：路况复杂是主要原因"
  → 修改第 3 项的修复内容

"保留第 2 项的对比"
  → 将第 2 项的 action 改为 "keep"
```

### 边界情况

**用户要求全部跳过**：
- 询问是否真的不需要任何修复
- 或者是否要调整检查标准

**用户提供新的修复建议**：
- 更新对应项的建议修改文本
- 记录用户自定义的修复

**模糊指令**：
- "第 3 项不太对"
- 询问用户具体想如何处理

### 错误恢复

- 用户反馈解析失败 → 再次询问，提供明确的选项
- 指定的项不存在 → 提示有效的项范围

## 步骤 5：执行修复

### 目标
安全、准确地执行文本修改。

### 修改策略

**从后往前修改**：
```typescript
const edits = decisions
  .filter(d => d.action === "fix")
  .sort((a, b) => b.line - a.line);  // 降序排列

for (const edit of edits) {
  edit(文件路径, {
    oldText: edit.oldText,
    newText: edit.newText
  });
}
```

**为什么从后往前**：
- 修改前面的行会导致后面的行号偏移
- 从后往前修改，前面的行号保持不变

### 精确匹配

**oldText 定位**：
- 使用完整的匹配文本（包括前后空格）
- 如果文本不唯一，使用行号辅助定位

**示例**：
```typescript
// 不好：可能匹配多处
oldText: "其实"

// 好：包含上下文，唯一匹配
oldText: "这个问题很复杂。其实我们可以从另一个角度来看。"
```

### 生成新文件

**不修改原文件**：
```typescript
// 所有修改完成后
const modifiedContent = read(原文件路径);
write(".agent/polish-output.md", modifiedContent);
```

**原因**：
- 保留原文件作为备份
- 用户可以对比修改前后

### 边界情况

**oldText 找不到**：
- 文件内容可能已经被外部修改
- 重新读取文件，更新 oldText
- 如果仍找不到，跳过该项并记录

**newText 引入新问题**：
- 第一版不检测
- 后续可以支持修复后再检查

**修改冲突**：
- 两个修改涉及同一行
- 按行号顺序执行，后面的修改可能失效
- 记录冲突并提示用户

### 错误恢复

- 修改失败 → 记录失败项，继续其他修改
- 文件锁定 → 提示用户关闭文件
- 权限不足 → 提示权限问题

## 步骤 6：生成修复报告

### 目标
提供清晰的修复总结和对比。

### 报告内容

**统计部分**：
```markdown
## 统计
- 总问题数：15
- 已修复：12
- 保留不改：3
- 修复类别：填充短语（8）、机械过渡（4）
```

**修复详情**：
```markdown
## 修复详情

### [1] 行 5:12 - 填充词：其实 ✓ 已修复
- **原文**：这个问题很复杂。其实我们可以从另一个角度来看。
+ **修改**：这个问题很复杂。我们可以从另一个角度来看。

### [2] 行 12:1 - 填充词：值得注意的是 ✓ 已修复
- **原文**：值得注意的是，这种方法在实践中效果显著。
+ **修改**：这种方法在实践中效果显著。

...

### [15] 行 156:8 - 问题定义对比 ✗ 保留
**原文**：问题不是技术，是沟通。
**理由**：用户确认保留，此处对比有强调作用
```

### 展示方式

**简要版**（推荐）：
```
修复完成！

- 总共 15 个问题
- 已修复 12 个
- 保留 3 个

输出文件：.agent/polish-output.md
详细报告：.agent/polish-final-report.md
```

**详细版**（用户要求时）：
- 展示完整报告内容
- 或展示几个关键修复的 diff

### 边界情况

**所有问题都被保留**：
- 告知用户没有执行任何修复
- 询问是否需要调整检查标准

**修复失败项**：
- 在报告中标注失败项
- 说明失败原因

### 错误恢复

- 报告生成失败 → 口头总结给用户
- 文件写入失败 → 在对话中展示报告内容

## 完整示例

### 用户输入

```
用户：帮我去除这段文字的 AI 味道

这个问题很复杂。其实我们可以从另一个角度来看。

值得注意的是，这种方法在实践中效果显著。

首先，我们需要分析问题。其次，要制定方案。最后，执行计划。
```

### Agent 执行

1. **获取输入**：写入 `.agent/polish-input.md`
2. **CLI 检查**：发现 4 个问题
3. **LLM 审查**：
   - "其实" → 修复
   - "值得注意的是" → 修复
   - "首先...其次...最后..." → 修复（error 级别）
4. **生成计划**：展示给用户
5. **用户确认**："确认"
6. **执行修复**：生成 `.agent/polish-output.md`
7. **生成报告**：展示统计和文件位置

### 修复后文本

```
这个问题很复杂。我们可以从另一个角度来看。

这种方法在实践中效果显著。

我们先分析问题，再制定方案，然后执行。
```

## 最佳实践

### 对于 Agent

1. **读取上下文充分**：不要只看问题本身，要读取前后文
2. **判断要有依据**：说明为什么修复或保留，引用规则定义
3. **不确定时询问**：不要强行做决定
4. **记录决策过程**：方便用户理解和审查

### 对于用户

1. **分段处理**：大文件建议分段检查和修复
2. **迭代优化**：第一轮修复后可以再检查一次
3. **保留原文件**：修复前备份重要内容
4. **理解规则**：阅读 patterns.md 了解什么是 AI 味道

---

## 未来改进方向

1. **自动迭代**：修复后自动再检查，直到无问题
2. **冲突检测**：检测修复引入的新问题
3. **批量处理**：一次处理多个文件
4. **自定义规则**：用户可以添加自己的规则
5. **学习优化**：根据用户反馈调整规则权重
