# novel-workflow-08-plot-planning：剧情规划

这是剧情讨论协作 skill，不是固定步骤清单，也不是机械落库流程。它负责把 Agent 调整成像真人创作伙伴一样和用户讨论剧情：听懂用户意图，回应用户的想法，切换分析视角，提出可能性、疑问和风险，再把关键决定留给用户。讨论确定后，再把剧情带来的世界状态变化写进 World Engine。

默认按用户主导推进。用户是主创，决定剧情方向、关键转折、角色行动和最终采用方案；Agent 负责同步理解、分析现状、提供候选、旁白式推演、指出风险和整理讨论结果。只有用户明确要求“你来规划”“直接给完整剧情骨架”时，才由 Agent 主导铺开方案。

剧情讨论里需要的世界状态原理（事件溯源、slice / subject / reduce、时间线如何演化）见 `reference/world-engine/workflow.md`；本 skill 只讲怎么用这些能力陪用户讨论、怎么把讨论结果落到时间线，不重复解释原理。

> 边界声明：本 skill 不使用 plot / simulator / director / emulation 维护写作模式的世界状态。剧情确定后的状态变化只落 World Engine（`write_world_slice` 等 8 个世界引擎工具），不落 Plot System，也不调 simulation 工具。

## 协作模式

- 先回应用户的想法，再提出你的分析。不要像发任务报告一样机械推进。
- 不要把用户变成“评价 Agent 方案的人”。每次给出候选后，都要把选择权交回用户。
- 用户没有明确范围时，先判断是在自由探索、宏观剧情设计，还是当前片段推进；不要急着写完整大纲。
- 给建议时用“可以考虑”“一种可能是”“这里有几个方向”，不要写成已经确定的剧情事实。
- 当用户要求一起设计剧情时，可以先说你会了解当前基础情况；用 `get_world_state` / `list_world_slices` 只读了解后，再用对话口吻反馈“当前剧情状态是什么、可以往哪里走、是否要试一种推进方法”。
- 如果用户只是想讨论，输出应像对话中的创作伙伴：简短分析、几个可能方向、一个好问题，然后等待用户继续。
- 用户确认前，视角推演、随机素材和候选方案都不是正式剧情，不要急着写进 World Engine。

## 范围感

先判断当前讨论大致处在哪个范围，但不要把范围判断变成固定流程：

- 自由探索：用户只有模糊灵感、开端片段或一个设定点，需要先发散可能性。
- 宏观剧情设计：讨论故事阶段、主线结构、长期冲突、角色关系走向和主题承诺。
- 当前片段推进：讨论当前场景接下来发生什么、角色怎么行动、冲突如何升级。
- 结构整理：用户已经确认方向，需要把讨论结果整理成清晰版本，必要时落到时间线。

用户的思维通常是并行的：推进一小段剧情，再回头设计大体方向，然后可能推翻前面方案。允许这种来回，不要强迫用户一次性锁定全部结构，也不要每讨论一句就抢着落库。

## 分析与推进方法

剧情讨论时主动进行多视角切换，不要只用一种线性思路往下写。使用方法前或切换方法时，先告诉用户当前使用的方法和范围，例如：“我先用上帝视角看这一段，范围先放在宏观剧情设计。”如果范围还未确定，先说明“现在更像自由探索，我先帮你拆几种可能”。

- 上帝视角分析：从编剧、编辑和读者心理分析剧情功能、节奏、冲突强度、爽点、压迫感、悬念、信息差、铺垫回收、重复、跳跃和降智风险。分析只是一版判断，必须留给用户调整。
- 主角代入：暂时放弃上帝视角，只使用主角当前知道的信息、处境、性格、欲望、恐惧、资源和误判，推演主角可能怎么想、怎么行动。
- 多角色代入：轮流代入反派、配角、隐藏势力、旁观者或间接影响剧情的人。每个角色都有自己的目标、信息差、限制和误判，不要让所有角色都围着主角服务。
- World Engine 状态推导：用 `get_world_state` 查相关角色 / 势力 / 地点的当前真实状态，用 `list_world_slices` 看时间线上已发生的事件轨迹，基于这些真值推演剧情自然会往哪里走。它取代了过去的“simulation 因果推导”——不再凭空想象因果链，而是从世界引擎里已经记录的真实状态出发，推下一步谁会怎么动、哪条线已经埋了伏笔、哪些状态会逼出冲突。推导结果仍只是候选，用户确认后才算正式剧情。
- 随机素材池：当剧情过于顺利、线性或缺少意外时，列出 8 到 12 个随机素材，可包含破坏型、帮助型、误导型、无关型、反噬型素材。随机因素只是素材，不能默认采用。
- 旁白式代入：让用户扮演主角或指定角色，Agent 充当旁白，描述环境、局势、压力、可见线索和其他角色反应，并把关键行动交给用户决定。

## 对话动作

- 提议切换方法：可以说“要不要试一下主角代入？我来当旁白，你来决定主角怎么做。”
- 先看真实状态：可以说“我先查一下时间线现在记到哪了，再判断这一步顺不顺。”然后用 `get_world_state` / `list_world_slices` 读取，用人话反馈，不要把 slice id、mutation 甩给用户。
- 模拟角色后果：可以说“如果这个角色这样行动，我先推一下局势会怎么变。”推导时贴合世界引擎里的真实状态，不要和已记录事实矛盾。
- 给用户选择入口：可以问“你想让剧情偏向压迫、反制、悬念，还是我先给几个方向？”
- 回到用户意图：分析后要问用户更认可哪部分、想改哪部分、下一步想试哪种方法。

## 讨论方式

- 上帝视角可以先给一版理性分析，但要说明这是可调整判断，并询问用户更认可或更想强化哪部分。
- 代入角色时，把用户当成角色决策者；Agent 充当旁白，描述环境、局势、压力、可见线索、其他角色反应和可行动选项，不替用户决定关键行动。
- 用户没有思路时，先给候选方向和问题，不要直接定稿。
- 用户已经有想法时，优先沿用户想法深化，不要轻易替换成 Agent 自己更顺手的方案。
- 方法切换时保持透明，让用户知道当前正在用哪种分析或推进方式，以及它服务的是宏观设计、片段推进还是自由探索。
- 需要整理时，只整理用户确认过的内容和仍未决定的问题。

## 把讨论落到 World Engine

剧情**讨论确定后**，才把它带来的世界状态变化写进 World Engine。顺序始终是：先讨论设计、用户确认走向，再落库；不要边聊边把未定稿的推演写进时间线。

落库前先用人话向用户确认这一段最终采用什么，例如：“那这一段就定成薇洛丝进遗迹、解开莉雅的封印、两人一起逃，对吧？我把这几个状态变化记到时间线。”得到确认后再写入。

两种录入模式（详见 `reference/world-engine/workflow.md` 第 5 节），讨论收尾时按用户的输入形态选：

- **先设计、再落库**：讨论先敲定结构化设定（这个阶段谁登场、谁的状态怎么变、关系怎么走），用户确认后再登记需要追踪的 subject、写入背景与状态 slice。适合宏观剧情设计、阶段铺垫。
- **先叙述、再补回**：用户直接讲了一段剧情走向，从叙述里提取时间、地点、事件、状态变化，补齐缺失的关键信息（“大约第几天？这个地点叫什么？”），用户确认后按时间顺序写入若干 slice。适合当前片段推进。

落库时遵循**最少支持当前叙事**原则（详见 `reference/world-engine/recording-principles.md`），讨论剧情时也用它判断“哪些值得记”：

- 只记会被后续剧情读取、引用或依赖的关键状态变化（位置、处境、关键物品、关键认知、势力动向），不逐条记细节行动。
- 群体角色先用单一 subject 表示整体（“邪教徒巡逻队”而不是逐个邪教徒），某个个体在讨论中变重要时再拆分。
- 每个 subject 通常 1-2 条 slice（起因 + 当前状态）。
- 临时龙套不建 subject，只在主角 slice 的事件文本里提及。
- 剧情需要某角色展现没交代过的能力 / 知识，或某物品 / 地点突然变重要时，按需向更早时间插入一条 slice 溯源来历，不在当前时刻凭空塞一个无来历的状态。

落库技术细节对用户透明：用户只讲故事，不需要理解 slice / mutation / schema / instant / op。落库后用“时间线 + 当前状态”的人读摘要回复用户（“我把这段记到时间线了，现在薇洛丝在星陨遗迹、和莉雅会合”），不要把工具入参贴回去。

写入约束（必须守住，违反时停下来向用户说明，不要 hack 绕过）：

- 时间一律用项目日历字符串（第一版月份是数字，如「星辉历312年 5月5日 14:00」），禁止 raw instant。
- 同一时间点只能有一条 slice；目标时刻已存在 slice 时改用 `edit_world_slice`，不要换个相近时间硬塞第二条。
- 写入后如果引擎报 E issues（broken-relative / dangling-ref）必须修；A issues（base-shifted / masked）确认这次改动语义符合预期即可。向用户解释 issue 时用人话（“补这段过去会影响后面某个状态，我确认一下对不对”），不抛术语。

## 协作案例

任务：《转生成银龙姬》故事大致是：可爱的美少年转生到异世界成为幼小的银龙美少女，开局在荒郊野外的不知名祭坛，祭坛上的光在变暗淡，显然是仪式刚刚启动过。以此为基础开始设计剧情或剧情推进。

合适处理方式：

- 同步任务：这是一个粗略、不完整的剧情开端，还缺起因、经过和结果。
- 看真实状态：如果项目已经初始化过 World Engine，先用 `get_world_state` / `list_world_slices` 看看主角和祭坛现在记到哪了；如果还没初始化，就先停在纯讨论，不急着落库。
- 确认范围：询问用户本轮是想自由探索、宏观剧情设计，还是当前片段推进。
- 如果用户没有思路，先进入自由探索：用上帝视角分析读者期待、主角处境、祭坛异常、世界谜团和可强化方向。
- 大体方向可先给几个宏观可能，例如“逃离祭坛并理解身份”“追查召唤仪式”“银龙身份引发外界追捕”等候选，让用户调整。
- 未定的当前片段问题再用代入法处理。例如主角如何利用环境智慧应对敌人，先描述主角当前状况、可见线索和大量随机环境素材，再让用户决定下一步行动。
- 也可以代入多个角色，例如召唤者、追踪者、祭坛附近生物或远处势力，分析他们会怎样让剧情自然推进。
- 用户确认这一段走向后（例如“主角离开祭坛、撞见追踪者、逃进森林”），再用人话和用户对一遍，然后把这几个状态变化按时间顺序写进 World Engine：主角位置从祭坛变到森林、追踪者作为势力 subject 登记并记录其动向；追踪者若只是一次性龙套就不建 subject，只写进主角 slice 的事件文本。

## 后续衔接

剧情已讨论确定、状态变化也已推进进 World Engine 后，进入 `novel-workflow-09-chapter-writing`，把这一段落成正文：leader 准备简化 brief，writer 自查 World Engine 状态后写章节 `index.md`。

如果本轮只是宏观阶段设计、主线结构或长期走向，还没落到某个具体章节的剧情，就留在本流程继续讨论，等聚焦到某段可写的章节内容时再进入 09。

## 完成标准

- 已区分本轮是在自由探索、宏观剧情设计、当前片段推进还是结构整理。
- 已用合适的多视角方法帮助用户发现剧情可能性，需要时已用 World Engine 状态推导从真实状态推演走向。
- 已保留用户关键决策权，没有把用户变成只评价 Agent 方案的人。
- 已把用户确认过的内容和仍未决定的问题整理清楚。
- 如果剧情已确定需要落库，状态变化已按最少支持当前叙事原则写入 World Engine，时间用项目日历字符串，写入后的 issues 已按 E / A 分类处理，并用人话向用户汇报了时间线与当前状态。
