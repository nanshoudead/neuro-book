## 读者（summary） 层级（区分大纲和 summary，大纲是作家的角度，summary 是读者的角度）

> 让 AI 维护三个不同的 summary 层级。（读者视角）（状态机的状态）

1. 全书 summary（高压缩）
2. 篇 summary（中压缩）
3. 章 summary（低压缩）

- 只有“发生了什么” → 高压缩
- 能回答“为什么会发生” → 中压缩
- 能看到“是怎么一步步变成这样的” → 低压缩
- 能“在脑中还原画面” → 极低压缩

简化为：Phase/Volume，Thread，Scene，Plot

Thread 和 Scene

人类是以 Scene 为单位构造剧情的。小说正文就是一个 Scene 列表。

一章包涵一个或多个 Scene，一个 Scene 由多个 Plot 组成。

为了剧情能便于被 AI 读取、管理、组织。引入了 Thread 的概念。

Thread 是一串以因果逻辑关系组织起来的 Scene 列表。Thread 是用于处理多线剧情的工具

小说正文的 Scene 列表由属于各个 Thread 的 Scene 构成。例如：Scene_1a(Thread1) -> Scene_1b(Thread1) -> Scene_2a(Thread2) -> Scene_1c(Thread1)

人类通常以 Scene 列表的视角进行工作。通常当 Scene 积累到一定数量后可以考虑分类 Scene。将一些 Scene 归类到 Thread。

注意 “主线（Main Thread）”。剧情系统应当以主线为中心。主线是带了 “主线” 标志的 Thread

为了避免主线过长，可以将长主线拆分成多个主线 Thread，。达到这样的结构：Main Thread1 -> Main Thread2 -> Main Thread3

遇到大型小说（例如《凡人修仙传》）的时候。主线 Thread 列表可能过长，可能有 500 多个。为了解决这个问题，引入 Phase/Volume

## 另一种视角：正文时间轴

除了树图和线程视角，还需要一种“正文时间轴”视角。

它的作用不是处理因果分叉，而是：

- 把一个 `StoryPhase` 下的多条 `Thread`
- 编排成正文展示顺序
- 直观看到哪些 Scene 在同一正文阶段并行出现

这个视角更接近动画、视频编辑软件里的横向时间轴：

- 每一条横向泳道代表一条 `Thread`
- 横轴先按正文顺序理解
- `Chapter` 更适合作为背景分段，而不是单独泳道
- 未挂章的 Scene 适合进入末尾草稿区

这一视角的重点是“怎么进正文”，不是“因果怎么分叉”。

## 另一种局部视角：单 Thread 侧边栏

还有一种更窄的工作形态：

- 一次只看一条 Thread
- Scene 自上而下排列
- 用底部 detail 面板补足当前 Scene 的结构化信息

这个视角更像 IDE 里的工具侧栏，而不是中央主画布。

它适合处理：

- 一条线内部的局部推进
- Scene 顺序检查
- 当前 Scene 的 purpose、plot、refs、writingTip 浏览
