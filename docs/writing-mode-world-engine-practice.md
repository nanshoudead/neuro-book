## Phase 1：理解 World Engine 文档体系

### 读取文档

按推荐顺序读取了 5 篇核心文档：

1. **workflow.md**：写作模式整体工作流
   - 三系统职责：World Engine（动态状态）、Lorebook（静态设定）、Manuscript（正文）
   - 技术细节对用户透明（不暴露 slice/patch/op 术语）
   - 两种录入模式：先设计再写 / 先叙述再提取
   - Leader-Writer 协作：Leader 推进世界状态，Writer 只读查询

2. **recording-principles.md**：最少支持当前叙事原则
   - 只记录会被后续剧情读取的事实
   - 群体角色先用单一 subject，需要时再拆分
   - 每个 subject 通常 1-2 条切面（起因 + 当前状态）
   - 临时龙套不建 subject，只在主角 events 里提及

3. **subject-lifecycle.md**：主体生命周期
   - 首写自动创建（C1 契约）：首次 patch 声明 `type` 字段即可
   - 切面是增量，状态由 reduce 得来
   - issues 分类：E issues（必须修）vs A issues（一次性提醒）

4. **schema-system.md**：schema 与 4-op 语义
   - kind 决定 op 语义：scalar/list/collection/object
   - 4 种 op：replace（绝对值）、increment（增减）、remove（移除）、append（追加）
   - ref 规则：`subject://id` 格式，目标必须存在

5. **calendar-system.md**：时间系统
   - Instant 是唯一时间真相源（bigint 秒数）
   - Calendar 负责显示转换（三种类型：simple/gregorian/custom）
   - Agent/HTTP 只接受日历字符串，不接受 raw instant

### 核心认知

- **World Engine 是唯一动态状态真相源**（不用 Plot、不用 simulation workflow）
- **技术细节对用户透明**（用户看到"时间线 + 角色状态"，不看到 patch JSON）
- **首写自动创建非常好用**（不需要单独 create 步骤）
- **删除切片可通过 Agent 工具完成**：先用 `execute_world_query` 的 `world.slices()` 获取 `sliceId`，再调用 `delete_world_slice`；HTTP/Workbench/facade 是其他入口。

## Phase 5：测试删除切片回退

### 任务

用户要求删除"5月15日初步对话"切片，回退到"召唤刚结束"的状态。

### 操作步骤

1. **查找切片 ID**：
   ```typescript
   const slices = await world.slices();
   const target = slices.find(s => s.title === '勇者们的初步对话');
   // 结果：sliceId = b394634a-5710-43a3-a2e3-ed35082c9cd6
   ```

2. **通过 facade 删除**：
   ```typescript
   import {worldEngineFacade} from 'nbook/server/world-engine';
   await worldEngineFacade.deleteSlice(projectPath, sliceId);
   ```

3. **验证**：
   - 返回 `issues: []`（无数据错误）
   - 时间线停在"复兴纪元488年5月10日凌晨：召唤仪式成功"

### 关键发现

- **Agent 工具已有删除能力**：`execute_world_query` 负责查 `sliceId`，`delete_world_slice` 负责物理删除切片
- **HTTP/Workbench/facade 也是删除入口**：脚本或维护场景仍可直接调用 `worldEngineFacade.deleteSlice`
- **删除是物理删除，不可恢复**：没有撤销机制

## 经验总结

### 成功的地方

1. **首写自动创建非常好用**：
   - 第一次写入 subject 时只需声明 `type` 和 `name`，schema 的 default 值自动应用
   - 不需要单独的"创建 subject"步骤，简化了流程

2. **lorebook 的 soul.md 结构很清晰**：
   - 调色盘三层（底色/主色调/点缀色）+ 二次解释，防止 AI 误读人设
   - 条件化习惯（if...then...）比固定行为更灵活
   - 元规则约束（不用上帝视角、不替其他角色做决定）是必要的安全边界

3. **LOD 机制的灵感很好**：
   - 用"角色能感知什么"来过滤世界细节，让信息控制更自然
   - 区分"纯装饰"、"伏笔"、"即将进入场景的事件"，让世界有层次感

4. **World Engine 的时间回退很干净**：
   - 删除切片后立即生效，不需要"撤销栈"或复杂的版本管理
   - 但要小心：删除是物理删除，不可恢复

### 遇到的问题

1. **删除工具需要明确使用边界**：
   - 先用 `execute_world_query` 的 `world.slices()` 找到目标 `sliceId`
   - 再用 `delete_world_slice` 物理删除；删除不可恢复，只用于剧情回退、修正错误切片或清理误写数据
   - 脚本或维护场景仍可通过 `worldEngineFacade.deleteSlice` / Workbench 删除

2. **切片粒度的权衡**：
   - 第一章 4 个切片，每个切片 5-10 分钟的戏内时间
   - 如果切片太细（每个对话一条切片），会导致切片数量爆炸
   - 如果切片太粗（整章一条切片），会丢失时间线细节
   - **当前实践**：一个"场景转换"或"关键状态变化"对应一条切片

3. **events 数组会无限增长**：
   - 每个角色的 `events` 是纯 append，不会自动清理
   - 长期运行后可能导致查询性能问题
   - **改进方向**：
     - 在 schema 层支持"只保留最近 N 条 events"
     - 或者引入"归档旧 events 到单独文件"的机制

4. **lorebook 与 World Engine 的同步问题**：
   - lorebook 的 `index.md` 是静态设定（"年龄：19岁"）
   - World Engine 的 `age` 是动态状态（会随时间变化）
   - 两者可能不一致
   - **当前实践**：lorebook 只写"外貌年龄"或"初始设定"，动态变化只在 World Engine 里

### 关注度等级的实践

在这次实战中，我们为 5 个角色分配了不同的星级，并按星级决定 backstory 切片数量：

| 角色 | 星级 | backstory 切片数量 | 实际使用 |
|------|------|-------------------|----------|
| **薇洛丝** | ★★★★★（绝对主角） | 建议 5-10 条 | 2 条（地球生活 + 转生） |
| **月涟** | ★★★★☆（核心主角） | 建议 5-8 条 | 3 条（泰拉背景 + 传送） |
| **格鲁什** | ★★★★☆（核心主角） | 建议 5-8 条 | 3 条（部落背景 + 拉入） |
| **艾莉娜** | ★★★★☆（核心主角） | 建议 5-8 条 | 3 条（学院背景 + 误入） |
| **维克托** | ★★★☆☆（重要配角） | 建议 2-4 条 | 4 条（出生 + 继承 + 决定召唤 + 当前） |

**关键发现**：
1. **薇洛丝的 backstory 偏少**：按 ★★★★★ 标准应该有 5-10 条完整生命线，但实际只有 2 条。原因是她是"转生者"，地球生活的细节可以后续按需补充。
2. **维克托的 backstory 偏多**：按 ★★★☆☆ 标准应该 2-4 条，实际用了 4 条（上限）。因为他是召唤者，需要解释"为什么召唤勇者"的完整动机链。
3. **当前剧情切片密度**：第一章 4 个切片，每个角色都有完整的 events 记录。如果严格按 ★★★★★ 的"每个场景转换都记录"标准，薇洛丝应该有更细的切片（如"苏醒 → 观察环境 → 听到声音 → 看到维克托"拆成 4 个切片）。

**改进方向**：
- 初始化时可以"少补 backstory，按需溯源"。不是一次性补满 5-10 条，而是先补 2-3 条核心事件，后续剧情需要时再插入更早的 backstory 切片。
- 关注度等级可以体现在"当前剧情切片密度"而不是"backstory 数量"上。薇洛丝作为 ★★★★★，她的每个心理变化、每次观察、每次决策都应该记录；而维克托作为 ★★★☆☆，只在他直接参与剧情时才记录。

### LOD 边界的踩坑

在设计第一章剧情时，我们借鉴了 LOD 机制的分层思路，但也遇到了边界问题：

**正确做法**（只描述环境/氛围）：
- ✅ "城堡大厅里传来脚步声，似乎有人在走动"
- ✅ "远处厨房飘来食物的香气"
- ✅ "窗外隐约能看到村庄的灯火"

**错误做法**（引入有名字的个体）：
- ❌ "老管家艾伯特带领勇者们前往客房"（如果后续不需要艾伯特出场，不应该给他名字）
- ❌ "女仆莉丝端着茶盘走进客房"（如果后续不与她对话，不应该单独提名）

**实际使用**：
- 第一章里我们提到了"老管家"，但没有给他名字，也没有建 subject。这是正确的——他只是"带路工具人"，不需要追踪状态。
- 如果第二章需要"老管家揭露家族秘密"，那时候再建 subject（★★☆☆☆ 次要配角），补 1 条 backstory（服侍布劳尔家族 30 年）。

**关键原则**：
- **不要提前为"可能会用到的角色"建 subject**。等到真正需要时再建，保持"最少支持当前叙事"原则。
- **LOD 里的个体用"一名"、"某个"、"一位"等不定冠词**，不要用具体名字。
- **一旦需要与主角直接对话、做出独立决策、追踪状态变化，立刻提升为 subject**。

### 切片粒度的权衡

第一章实战中，我们用了 4 个切片覆盖 5 小时戏内时间（488年5月10日 00:05 ~ 05:00）：

| 切片 | 时间点 | 戏内时长 | 内容 |
|------|-------|---------|------|
| 1 | 00:05 | ~10 分钟 | 初次对话：身份与来历 |
| 2 | 00:30 | ~30 分钟 | 前往客房：初步探索 |
| 3 | 03:00 | ~1 小时 | 深夜对话：建立初步信任 |
| 4 | 05:00 | 瞬间 | 黎明决定：临时共识 |

**粒度分析**：
- 平均每个切片覆盖 1-2 小时戏内时间
- 切片 1 和切片 4 是"关键对话"，时间短但信息密度高
- 切片 2 是"移动 + 观察"，时间较长但状态变化少
- 切片 3 是"深入交流"，时间长且状态变化多（建立信任）

**如果是战斗场景**，同样 5 小时可能需要 10-15 个切片：
- 每个回合一条切片（"主角挥剑 → 敌人格挡 → 主角受伤 → 队友支援"）
- HP、MP、位置、状态效果（中毒、眩晕）频繁变化
- 每个角色的心理状态（恐惧、愤怒、冷静）也需要记录

**如果是赶路场景**，同样 5 小时可能只需要 1-2 个切片：
- "从城堡出发，沿着山路前往边境哨所，途中遭遇小股盗匪，轻松击退，傍晚抵达哨所"
- 只记录起点、终点、路上的关键事件，不记录每一步

**关键原则**：
- **动态调整粒度**：主角视角附近细，远处粗；新事件细，旧事件粗；战斗细，日常粗。
- **不要固定"每 X 分钟一条切片"**：根据信息密度和状态变化频率决定。
- **场景转换是天然的切片边界**：离开祭坛 → 前往客房 → 到达客房（3 个切片）。
